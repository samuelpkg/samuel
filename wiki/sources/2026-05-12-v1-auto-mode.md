## Key claims

### Data model (`auto.go`)

- `AutoDir = ".claude/auto"`. Schema version `AutoSchemaVer = "1.0"`. (`auto.go:14-21`)
- **AutoPRD** wraps a single JSON file: `Project` (name, description, source_prd, timestamps), `Config` (max_iterations, quality_checks, ai_tool, sandbox), `Tasks []AutoTask`, `Progress`. (`auto.go:73-79`)
- **AutoConfig** carries sandbox config + pilot mode toggles + progress rotation thresholds. (`auto.go:91-106`)
- **AutoTask** fields: `ID`, `Title`, `Description`, `Status`, `Priority`, `Complexity`, `ParentID`, `DependsOn`, `FilesToCreate`, `FilesToModify`, `Guardrails`, `CompletedAt`, `CommitSHA`, `Iteration`, `Source`. (`auto.go:116-132`)
- Status enum: pending / in_progress / completed / skipped / blocked. Priority: critical > high > medium > low. Complexity: simple / medium / complex. Sources: manual / prd / pilot-discovery.
- Loop status: not_started / running / paused / completed / failed.
- **Custom UnmarshalJSON for AutoTask** handles numeric IDs from AI tools that emit `"id": 1` instead of `"id": "1"`. Falls back to string parse. (`auto.go:137-179`) ŌĆö defensive against unreliable AI output.
- **Atomic save**: write to `<path>.tmp`, then rename. Prevents partial writes corrupting prd.json mid-iteration. (`auto.go:256-264`)
- Supported AI tools allowlist: `claude`, `amp`, `cursor`, `codex`. (`auto.go:281-283`)

### Loop (`auto_loop.go`)

The main loop:

```
for i = 1..MaxIterations:
    prd = LoadAutoPRD(.claude/auto/prd.json)        // reload each iteration ŌĆö agent may have edited it
    if prd.GetNextTask() == nil: return nil
    
    notifyIterStart(...)
    PrepareProgressContext(projectDir)               // regen progress-context.md, rotate progress.md
    GenerateProjectSnapshot(projectDir)              // regen project-snapshot.md
    GenerateTaskContext(projectDir, prd, isDiscovery)// regen task-context.md
    
    err = InvokeAgent(cfg)
    if err:
        consecutiveFailures++
        if consecutiveFailures >= MaxConsecFails: abort
    else:
        consecutiveFailures = 0
    
    sleep PauseSecs
```

- `PauseSecs` (default 2) and `MaxConsecFails` (default 3) configurable via env `PAUSE_SECONDS`, `MAX_CONSECUTIVE_FAILURES`. (`auto_loop.go:29-55`)
- **`InvokeAgent` allowlist check first** ŌĆö refuses to exec if `cfg.AITool` isn't in `GetSupportedAITools()`. Defense against agent injection via tampered prd.json. (`auto_loop.go:105-110`)
- Three sandbox paths: `SandboxDockerSandbox`, `SandboxDocker`, `SandboxNone`. (`auto_loop.go:112-119`)
- Docker `run` args: `--rm --init -i --user $UID:$GID -v <projectDir>:/workspace -w /workspace <envs> <image> <aiTool> <agentArgs>`. (`auto_loop.go:206-217`)
- Image validation via `IsValidSandboxImage` before `docker run`. (`auto_loop.go:160-164`)
- `requiresPromptContent(aiTool)` returns true for claude/codex ŌĆö for these, the prompt FILE content is read on host and passed as content arg (because the file isn't reachable inside the container without an extra mount). (`auto_loop.go:174-183`)
- Callback hooks: `OnIterStart(iter, type)`, `OnIterEnd(iter, err)` ŌĆö used by the CLI to render progress. (`auto_loop.go:23-25`)

### Pilot mode (`auto_pilot.go`)

- `InitPilotPRD` creates an empty prd configured for pilot mode (PilotConfig with discover_interval, max_discovery_tasks, focus). (`auto_pilot.go:54-86`)
- `ShouldRunDiscovery` decides at each iteration whether to run a discovery (no-code) pass:
  - no pending tasks ŌåÆ discover
  - first iteration (lastDiscoveryIter == 0) ŌåÆ discover
  - interval elapsed since last discovery ŌåÆ discover
  - pending count < `MinPendingTasksForDiscovery` (default 2) ŌåÆ preemptive discover
- Defaults: `DefaultPilotIterations = 30`, `DefaultDiscoverInterval = 5`, `DefaultMaxDiscoveryTasks = 10`, `MaxEmptyDiscoveries = 2`. (`auto.go:29-38`)

### Task selection (`auto_tasks.go`)

- `GetNextTask` returns the highest-priority pending task whose `DependsOn` are all completed/skipped. Tie-break: lower ID first.
- Priority ranks: critical=0, high=1, medium=2 (default), low=3.
- `ValidateAutoPRD` checks: schema version present, project.name present, task IDs unique and non-empty, statuses valid, dependency IDs resolve to known tasks.
- `RecalculateProgress` runs on every save ŌĆö recomputes `TotalTasks`/`CompletedTasks` and bumps Status to `LoopStatusCompleted` when all done.
- `NextAvailableID` walks tasks' top-level integer prefixes and returns `max+1` as a string (`"1"` start, supports dotted notation like `"2.1"` without bumping).

### Prompts (`auto_prompt.go`, `auto_discovery_prompt.go`)

Both prompts emphasize **context efficiency** ŌĆö token discipline is a first-class methodology concern.

**Implementation prompt** says explicitly (paraphrasing):
- Read `.claude/auto/task-context.md` FIRST. Do NOT read full prd.json.
- Read `.claude/auto/progress-context.md` for prior learnings.
- Read only files listed in `files_to_modify`. Don't explore.
- Read `CLAUDE.md` only if needed.
- Update prd.json directly for status changes.
- Conventional commit format, include task ID in message.
- Document learnings to `.claude/auto/progress.md`: `[timestamp] [iteration:N] [task:ID] LEARNING: ...`

**Discovery prompt** says:
- Read `project-snapshot.md`, `task-context.md`, `progress-context.md` FIRST. Do NOT manually scan tree, run find/ls, or grep for TODOs.
- Use grep over file-reads.
- Read at most 10 source files per discovery iteration.
- Skip files already covered by pending tasks or already analyzed (per progress-context).
- Generate ONLY atomic tasks (Ōēż5 files each).
- Do NOT make code changes ŌĆö only update prd.json + progress.md.
- Priority order: security > tests > quality > docs > perf > refactor.
- Each new task must have `source: "pilot-discovery"`.

Both prompts include the same guardrails: functions Ōēż50 lines, files Ōēż300 lines, tests required, conventional commits.

Both prompts reference **"Ralph Wiggum methodology"** by name (auto_prompt.go:14).

### Pre-computed context files

Three artifacts regenerated each iteration to keep the LLM token-thrifty.

- **`task-context.md`** (`auto_task_context.go`):
  - Implementation mode: current task's full detail (ID, title, priority, complexity, description, files_to_modify, files_to_create, dependencies) + last 5 recent completions for context.
  - Discovery mode: summary table (counts), pending tasks list, completed tasks (titles only ŌĆö save tokens), files already covered by pending tasks (don't re-task them).

- **`progress-context.md`** (`auto_progress_context.go`):
  - Summary: "N iterations completed, M tasks done".
  - Last 50 `LEARNING:` entries.
  - Last 10 `COMPLETED:` entries.
  - Deduplicated `EXPLORED:` file paths (skip these during discovery).
  - Defaults configurable via `AutoConfig.ProgressMax{Learnings,Completed,Lines}`.
  - **`RotateProgressIfNeeded`**: when `progress.md` exceeds 500 lines, archives older half to `progress-archive-<timestamp>.md`, keeps recent half.

- **`project-snapshot.md`** (`auto_snapshot.go`):
  - Source-file inventory (max 200 files, source extensions only: go/py/js/ts/rs/java/rb).
  - Test gaps (Go source files lacking `_test.go`).
  - 15 largest files (refactor candidates).
  - TODO/FIXME/HACK counts per file (top 30 by count).
  - Last 10 lines of `git log --oneline`.

### Progress format (`auto_progress.go`)

Append-only line log:
```
[2026-05-12T10:23:45Z] [iteration:5] [task:1.2] LEARNING: <message>
```

Entry types: `STARTED`, `COMPLETED`, `ERROR`, `LEARNING`, `QUALITY_CHECK`, `COMMIT`. `AppendProgress(path, entry)` opens with `O_APPEND|O_CREATE|O_WRONLY`. `ReadProgressTail(path, lines)` for tail reads.

### PRD conversion (`auto_convert.go`)

Bridges human-authored markdown ŌåÆ machine-readable prd.json.

Task line format (matched by regex):
```
- [ ] 1.0 Parent task title [~3,000 tokens - Medium]
  - [ ] 1.1 Sub-task description [~2,000 tokens - Simple]
```

- Indented tasks become children of the last unindented task.
- Child tasks get `ParentID` + `DependsOn = [parent]`.
- Checkbox `[x]` ŌåÆ status completed; `[ ]` ŌåÆ pending.
- Complexity from bracket suffix; default medium.
- `ConvertMarkdownToPRD(prdPath, tasksPath)` extracts title from PRD's H1, slugifies for `project.name`.
- `FindTasksFile` discovers `tasks-<prd-filename>.md` next to a PRD.

## Assessment

- **Credibility**: high.
- **Coverage**: this is the deepest pass yet. Auto-mode is the methodological centerpiece. Other workflows orbit it.
- **Quality of the v1 implementation**: high. The token-discipline approach (pre-computed context, "read these files first, don't grep") is a real innovation. The atomic-save + tool-allowlist + consecutive-failure-abort safety net is well-considered. The data model handles AI-output messiness (custom UnmarshalJSON for numeric IDs).

## v2 implications

### `#rescue` (high confidence)

- **prd.json data model.** Keep this verbatim. It's already a well-designed state machine.
- **Pre-computed context pattern.** Move into [[concepts/pre-computed-context]]. This is the kind of opinionated convention that belongs in the framework, not a plugin.
- **Two-mode design (implementation + discovery).** See [[concepts/pilot-mode]].
- **Ralph Wiggum methodology grounding.** Cite the source. See [[concepts/ralph-wiggum-methodology]].
- **Safety primitives**: atomic save, AI-tool allowlist, image regex validation, consecutive-failure abort, progress rotation.
- **Custom UnmarshalJSON for resilience** to AI output quirks. Generalize as a util ŌĆö every AI-emitted JSON field needs this.

### `#refactor` (carry forward with changes)

- The hardcoded `switch cfg.Sandbox` in `InvokeAgent` becomes the v2 OCI-runtime detect (Podman > Docker > ŌĆ”) per [[concepts/plugin-format]].
- The supported-AI-tools allowlist (`claude`, `amp`, `cursor`, `codex`) is hardcoded ŌĆö should be driven by installed agent-adapter plugins. See [[concepts/multi-agent-support]].
- The prompt strings are Go literals in `auto_prompt.go` / `auto_discovery_prompt.go`. v2 should externalize as template files (still built-in defaults, but overridable per project).
- The pre-computed context generators (snapshot, progress-context, task-context) are good candidates for hook extension points ŌĆö plugins can attach additional analyzers.

### `#drop`

- Hardcoded source-file extensions in `auto_snapshot.go:91-99`. Drive from installed language plugins instead.
- Hardcoded guardrail text ("functions Ōēż50 lines, files Ōēż300 lines") embedded in prompts. Source from `samuel.toml` `[guardrails]` block.

### `#open`

- Prompt customization model: built-in template + per-project overrides via `samuel.toml`, or full prompt-template plugins? (Likely: built-in default + samuel.toml overrides for common knobs, full plugin replacement for power users.)
- Hook points in the auto-mode loop:
  - `before:iteration`, `after:iteration`
  - `before:agent.invoke`, `after:agent.invoke`
  - `context.snapshot`, `context.progress`, `context.task` ŌĆö plugins can replace or augment
  - `iteration.discovery.gate` ŌĆö plugins can influence `ShouldRunDiscovery`
- Is "Ralph Wiggum" the right name for v2 marketing or should we rename? (Internal name OK; external maybe "autopilot" or "auto-loop".)
