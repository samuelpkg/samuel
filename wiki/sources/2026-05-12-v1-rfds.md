## Key claims

### The RFD process

Inspired by [Oxide Computer Company's RFD system](https://oxide.computer/blog/rfd-1-requests-for-discussion), which derives from IETF's RFC tradition.

**States** (state machine):

```
Prediscussion → Ideation → Discussion → Published → Committed
                    ↓           ↓
                    └─────→ Abandoned
```

- **Prediscussion / Ideation**: private in `.claude/rfd/` (author still forming idea).
- **Discussion / Published / Committed / Abandoned**: public in `docs/rfd/`.

**Frontmatter** (every RFD):

```yaml
---
rfd: "0001"
title: Progressive Disclosure Architecture
authors:
  - name: ar4mirez
state: Committed
labels: [architecture, documentation, token-efficiency]
created: 2025-01-15
updated: 2026-01-15
discussion: null              # link to discussion thread, optional
related_prd: null             # link to PRD if RFD became a PRD
---
```

**Structure** (every RFD body):

1. Summary
2. Problem Statement (+ Requirements + Constraints)
3. Background (research, prior art, existing patterns)
4. Options Considered (each with Pros, Cons, Effort)
5. Decision (which option, why)
6. Implementation plan
7. Outcome (post-implementation reflection)

### RFD vs PRD

| RFD | PRD |
|---|---|
| Explore options, build consensus | Define implementation |
| "Why" and "What options" | "What" and "How" |
| Early ideation | Ready to implement |
| Decision documented | Task list generated |

Flow: **Idea → RFD (decide) → PRD (plan) → Tasks → Code**.

### The four committed RFDs

#### RFD 0001 — Progressive Disclosure Architecture (2025-01-15) #v2-warning

**Problem**: CLAUDE.md was 500 lines / ~4,000 tokens. Violated its own token efficiency principles.

**Decision**: Core CLAUDE.md (~280 lines target) + on-demand context loading from `.claude/`. Token budget: ~2,000 tokens for system instructions.

**Key insights** (worth quoting):
- Anthropic recommends 150-200 lines for instruction files.
- "Automation > Documentation" — ESLint config > prose about linting.
- "Constraints > Aspirations" — "No functions >50 lines" > "Write clean code."
- "Right altitude matters" — specific boundaries, not implementation steps.

**Cautionary tale**: v1 made this decision **18 months ago** with a target of ~280 lines. The CLAUDE.md template I read in pass 9 is now **474 lines**. The template drifted back to bloat. v2 must enforce the limit with CI, not just a target.

#### RFD 0002 — Idiomatic Go Project Layout (2026-01-14)

**Problem**: Samuel CLI was at `packages/cli/` (Node.js monorepo pattern). Non-idiomatic for Go.

**Decision**: Standard Go layout — `go.mod` at repo root, `cmd/`, `internal/` at root.

**Key insight**: "A framework promoting best practices for AI-assisted development should itself follow language-idiomatic patterns. Using Node.js conventions for a Go project undermined credibility."

**Status**: implemented. v1's current layout follows this.

#### RFD 0003 — Composable CLI Commands Over Overloaded Flags (2026-01-14)

**Problem**: 7 commands originally. Missing capabilities: search, preview, config, diff.

**Decision**: Add 4 new focused commands (`search`, `info`, `config`, `diff`) instead of extending existing commands with flags. Unix philosophy: each command does one thing well.

**Status**: implemented. v1's command tree (pass 6) includes all four.

#### RFD 0004 — Smart Interactive Mode via Flag Detection (2026-01-14)

**Problem**: `samuel init --template full my-project` hung at confirmation prompt despite all values being provided.

**Decision**: Auto-detect when flags are provided → skip prompts. **Not** an explicit `--non-interactive`. **Not** TTY detection. Flag detection.

**Status**: implemented.

**Pattern worth porting**: any v2 command with interactive prompts should skip them when CLI flags fully specify the operation.

### The RFD process IS the documentation

Every major v1 decision has an RFD. The RFDs document:
- What the alternatives were (often 3-4 options per decision).
- Why each was rejected.
- What the chosen path required (effort, scope, follow-ups).
- What actually happened (post-hoc outcome).

This is **institutional memory in a queryable form.** Future maintainers can reconstruct the why behind any major choice. Compare to "we did it this way because reasons" tribal knowledge.

## Assessment

- **Credibility**: high.
- **Quality of the RFDs**: professional. Real options analysis, not rubber-stamp.
- **Value to v2**: the RFD process itself is a `#rescue` flagship pattern. The individual decisions in 0001-0004 are all relevant to v2 design.

## v2 implications

### `#rescue` — the RFD process

Port verbatim:

- **State machine** (Prediscussion → Ideation → Discussion → Published → Committed → Abandoned).
- **YAML frontmatter** shape.
- **Body structure** (Summary, Problem, Background, Options, Decision, Outcome).
- **`rfd-index.yaml`** as single source of truth (or `rfd-index.toml` per [[entities/config-format]]).
- **Public/private split** by state (private in `.samuel/rfd/`, public in `docs/rfd/`).
- **`create-rfd` plugin** lives in the starter pack ([[synthesis/v2-skill-migration-plan]]).

This is the methodology that produces good architecture decisions. Don't reinvent.
