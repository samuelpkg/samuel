# v1 Config + Sync

Ingest pass 2. Covers project/global config, the per-folder CLAUDE.md sync, the tarball download path, file extractor, and the v1 sandbox layer.

## Files

- `samuel_v1/internal/core/config.go` — `Config`, `InstalledItems`, `GlobalConfig`, `AutoYAML`
- `samuel_v1/internal/core/sync.go` — per-folder CLAUDE.md/AGENTS.md generator
- `samuel_v1/internal/core/downloader.go` — tar.gz fetch from GitHub repo
- `samuel_v1/internal/core/extractor.go` — file/dir copy with backup + path-traversal protection
- `samuel_v1/internal/core/docker.go` — sandbox modes + multi-agent runner
- `samuel_v1/samuel.yaml` — v1's own dogfood config

## Key claims

### Config (`config.go`)

- File name: `samuel.yaml` (primary) or `.samuel.yaml` (hidden fallback). YAML, `gopkg.in/yaml.v3`. (`config.go:13-17`, `63-72`)
- `Config` fields: `Version`, `Installed` (Languages/Frameworks/Workflows/Skills), `Registry`, `Auto`. (`config.go:20-25`)
- `AutoYAML`: `enabled bool`, `ai_tool string`, `max_iterations int`, `quality_checks []string`. (`config.go:28-33`)
- **Four parallel installed lists** — same triple-bookkeeping problem as `registry.go`. `AddLanguage` adds to both `Installed.Languages` and `Installed.Skills` via `LanguageToSkillName`. Same for Framework, Workflow. (`config.go:156-185`)
- Explicit migration methods (`MigrateLanguagesToSkills`, etc.) backfill the Skills mirror for older configs. (`config.go:235-278`)
- `GlobalConfig` at `~/.config/samuel/`. Cache at `~/.config/samuel/cache/`. **Path hardcoded** — does not use `os.UserConfigDir()` (would break on Windows). (`config.go:280-318`)
- Default registry URL: `https://github.com/ar4mirez/samuel` (set in `registry.go:6`). (`config.go:53`)
- `samuel config get/set` driven by `ValidConfigKeys` allowlist. Eleven supported keys. (`config.go:321-393`)
- v1 dogfood `samuel.yaml` (`samuel_v1/samuel.yaml`): Go installed, auto enabled with `claude`, quality checks = `go build`, `go test`, `go vet`.

### Sync — per-folder CLAUDE.md (`sync.go`)

**This is not skill-sync.** It walks a project tree and generates auto-managed CLAUDE.md + AGENTS.md files in subdirectories.

- `SyncFolderCLAUDEMDs` walks `opts.RootDir` with `filepath.WalkDir`. Respects `MaxDepth`, skips hidden dirs and a known skip set (`node_modules`, `vendor`, `target`, `dist`, etc.). (`sync.go:188-249`)
- For each directory, `AnalyzeFolder` counts file extensions, detects test files, identifies key files (main.go, package.json, Dockerfile, ...). (`sync.go:296-340`)
- `GenerateCLAUDEMDContent` renders a markdown stub: Purpose (from `folderPurposeMap`), Languages (counts), Key Files, Testing. (`sync.go:343-378`)
- **User-customization preservation**: file content is checked for `<!-- Auto-generated by Samuel` marker. Auto-generated files get rewritten. User-edited files are skipped unless `--force`. (`sync.go:38-39`, `252-293`, `380-383`)
- Writes to **both** `CLAUDE.md` and `AGENTS.md` in every directory — mirror for cross-tool compat. (`sync.go:241-243`)
- Extension → language map covers 30+ languages including TOML. (`sync.go:42-85`)
- Folder name → purpose map covers ~35 conventional names (cmd, internal, pkg, api, web, models, handlers, services, ...). (`sync.go:88-129`)

### Downloader (`downloader.go`)

- Fetches `https://github.com/ar4mirez/samuel` tar.gz archive for a given version. Uses internal `github` client. (`downloader.go:40-106`)
- "dev" = `main` branch, cache is cleared on each request. Released versions = tag, cached at `~/.config/samuel/cache/samuel-<version>/`. (`downloader.go:42-52`)
- `extractTarGz` is the security-critical extractor. (`downloader.go:126-196`)
  - Path traversal: rejects entries whose resolved path is outside `dest`. (`downloader.go:144-148`)
  - Size cap: `MaxExtractedFileSize = 100 MB` per file, defense against decompression bombs. (`downloader.go:17`, `167-178`)
  - Symlink validation: rejects absolute symlink targets, rejects relative targets that resolve outside `dest`. (`downloader.go:180-218`)
- Cross-device `os.Rename` fallback via recursive copy. (`downloader.go:98-103`, `220-267`)
- `ClearCache`, `GetCacheSize`, `CheckForUpdates`, `DownloadFile`, `GetLatestVersion` exposed as ops.

### Extractor (`extractor.go`)

- Copies files from a cached download (`<cache>/template/<path>`) to a project destination (`<project>/<path>`). Strips `TemplatePrefix = "template/"`. (`extractor.go:36-80`)
- `force` controls overwrite. Without `force`, existing files are skipped. (`extractor.go:82-114`)
- `ExtractAll` walks the whole `template/` tree. (`extractor.go:151-190`)
- `validateContainedPath` is the path-traversal guard used by every public method. (`extractor.go:216-224`)
- Backup/restore: `BackupFile(path, backupDir)` and `RestoreBackup(backupDir)` for safe rollback. (`extractor.go:288-352`)
- Convenience helpers: `WriteFile`, `ReadFile`, `RemoveFile`, `FileExists`, all path-traversal-checked. (`extractor.go:244-285`)

### Docker / sandbox (`docker.go`)

This is the seed of v2's sandbox model. v1 already does most of it.

- Three modes: `SandboxNone`, `SandboxDocker`, `SandboxDockerSandbox` (Docker Desktop Sandbox plugin). (`docker.go:14-23`)
- **Multi-agent support**: `DockerSandboxRunConfig.Agent` accepts `claude`, `codex`, `copilot`, `gemini`, `kiro`. Default `claude`. (`docker.go:21-23`, `39-45`)
- Env var allowlist (`aiToolEnvVarNames`): `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `AMP_API_KEY`, `AI_TOOL`, `PAUSE_SECONDS`, `MAX_CONSECUTIVE_FAILURES`, `TERM`. Only forwarded if set on host. (`docker.go:27-36`, `92-101`)
- Image validation regex: `^[a-zA-Z0-9][a-zA-Z0-9._\-/]*(:[a-zA-Z0-9._\-]+)?(@sha256:[a-f0-9]{64})?$`. Rejects shell metachars, absolute paths, leading dots. (`docker.go:60-75`)
- `CheckDockerAvailable` / `CheckDockerSandboxAvailable` — both check `PATH` and verify daemon/plugin. 5s context timeout. (`docker.go:77-119`)
- `BuildDockerSandboxArgs` constructs `docker sandbox run [--name] [--template] <agent> <workdir> [-- agent-args]`. (`docker.go:121-151`)
- **Per-agent prompt translation** (`GetAgentArgs`): (`docker.go:156-180`)
  - `claude`: read prompt file, pass content with `-p` + `--dangerously-skip-permissions`
  - `codex`: read prompt file, pass content after `--dangerously-bypass-approvals-and-sandbox exec`
  - `amp`: `--prompt-file <path>`
  - default: pass prompt path as positional arg
- Mount point: `DockerContainerMount = "/workspace"`. Default image: `node:lts`.

## Assessment

- **Credibility**: high.
- **Coverage**: this pass illuminates two big things — (a) v1 already has a multi-agent sandbox, and (b) v1 has a useful per-folder CLAUDE.md generator that's worth keeping. Both alter the v2 design direction (less to invent, more to refine).

## v2 implications

### `#rescue`

- **Sandbox abstraction in `docker.go`** — the structure (agent, workdir, env allowlist, prompt translation) is the right shape. Generalize from "docker / docker-sandbox" to "OCI runtime detection" per [[concepts/plugin-format]].
- **Multi-agent support** — already covers Claude, Codex, Copilot, Gemini, Kiro. The per-tool prompt translation pattern is exactly the cross-tool primitive v2's "thin core" should ship. See [[concepts/multi-agent-support]].
- **Per-folder CLAUDE.md generator** (`sync.go`) — useful as-is. See [[concepts/per-folder-context]].
- **Tar/extract security** — path traversal, symlink validation, size cap. Port directly to whatever transport v2 uses.
- **Auto-gen marker preservation** — `<!-- Auto-generated by Samuel` is a good convention. Reuse for any v2-managed files.

### `#drop`

- **YAML config** — switch to TOML per `#v2-decision`. The `samuel.yaml` schema is small enough to retranslate trivially.
- **Four-parallel-lists structure** in `Installed` — collapse to a single `installed.plugins` list with kind/category as metadata.
- **Hardcoded `~/.config/samuel`** — use `os.UserConfigDir()` so Windows works.
- **`template/` directory + cache prefix coupling** — vestigial under the embed model, and incompatible with the plugin model. Drop.
- **Tarball-from-GitHub-repo install** — replace with per-plugin fetch (Git repo, OCI image, WASM module).
- **Migration methods** (`MigrateLanguagesToSkills` etc.) — no migration in v2 by user decision (clean break).

### `#open`

- Auto-mode config: `ai_tool`, `max_iterations`, `quality_checks` belong in core config or in the auto-mode plugin's config? (resolve in pass 3)
- Sandbox mode default: `none` (v1 default) keeps "skill-only" workflows lightweight, but the v2 thesis is "sandboxed by default". Decide for v2.
- Should `samuel.toml` be flat or have separate sections per kind (`[plugins.skills]`, `[plugins.commands]`)?
