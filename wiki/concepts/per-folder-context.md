# Per-folder context

Each directory in a project gets its own `AGENTS.md` with a short description of what lives there. AI agents load this file when working in the directory, so they have local context everywhere — not just at the project root.

In v1, Samuel writes both `CLAUDE.md` and `AGENTS.md` as a mirror. **v2 writes AGENTS.md only by default**; CLAUDE.md and other tool-specific variants come from translator plugins. See [[concepts/agents-md-primary]] for the rationale.

Implementation in v1: [[entities/sync-claude-md]].

## Why it works

Coding assistants increasingly support per-folder context files. A short note like "this directory contains HTTP handlers" / "this directory contains data migrations" is much higher signal than re-deriving from filenames at every prompt.

The autogen marker pattern:

```
<!-- Auto-generated by Samuel
```

is the trust contract. With the marker, Samuel can rewrite freely. Without it, the file is user-owned and Samuel leaves it alone.

## v1's two-file mirror (changed in v2)

v1 writes both `CLAUDE.md` and `AGENTS.md` with the same content as a defensive cross-tool gesture. v2 collapses this:

- **v2 framework writes**: AGENTS.md only.
- **Tool-specific files**: optional translator plugins. A `claude-translator` plugin mirrors AGENTS.md → CLAUDE.md verbatim. A `cursor-translator` plugin emits `.cursor/rules/*.md` in the format Cursor expects. Etc.
- **Rationale**: see [[concepts/agents-md-primary]].

## Generator inputs

- File extensions (counted, mapped to language names)
- Folder name (mapped to a purpose)
- Key files detected (entry points, manifests, Dockerfile)
- Test presence

All from heuristic maps in `sync.go`. Reasonable defaults; v2 should make them user-extensible (`samuel.toml` overrides).

## v2 decision #v2-decision

- **Built-in core capability.** Part of the framework, not a plugin.
- **AGENTS.md only.** Tool-specific files are translator plugins. See [[concepts/agents-md-primary]].
- **Sync runs as hooks AND a top-level command.** Same code path, two trigger surfaces:
  - Hooks: `samuel init`, `samuel install <plugin>`, optionally per-iteration in `samuel run`.
  - Command: `samuel sync` for manual `--dry-run` / `--force` runs.
- **Heuristic maps** stay default-friendly but become user-overridable via `samuel.toml`.

## Open

- Marker syntax — keep `<!-- Auto-generated by Samuel` or switch to something more explicit / structured (e.g. frontmatter `samuel: auto-generated`)?
