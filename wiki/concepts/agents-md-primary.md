# AGENTS.md primary

v2 writes **AGENTS.md** as the canonical project / per-folder agent-instruction file. Tool-specific variants (CLAUDE.md, Cursor rules, Continue rules, etc.) are **plugin-driven translations**, not framework defaults.

## Why

Samuel's positioning is "Rails for coding assistants" (plural). Built-in primitives should be cross-tool. Hardcoding Claude Code's filename as the primary is a v1-era choice — useful at the time, but ties the framework's default output to one vendor.

The [Agent Skills](https://agentskills.io) standard already uses AGENTS.md as the cross-tool canonical name. OpenAI Codex, Cursor, Claude Code, GitHub Copilot all read it (or are converging on it). Writing AGENTS.md as primary means every agent has a consistent landing surface.

## What changes from v1

v1 writes **both** files with identical content as a mirror — see [[entities/sync-claude-md]] and [[concepts/per-folder-context]]. Helpful for v1's user base (mostly Claude Code), but redundant: two files with the same bytes.

v2 inverts:

- **Framework writes**: `AGENTS.md` (root + per-folder).
- **Optional plugin**: a `claude-translator` plugin watches AGENTS.md and emits CLAUDE.md as a verbatim copy. Same for `cursor-translator`, `continue-translator`, etc. Each plugin owns the per-tool filename and any tool-specific transformations (e.g., Cursor's `.cursor/rules/` directory format).
- **Result for the user**: `samuel init` installs AGENTS.md by default. If they want CLAUDE.md, they `samuel install claude-translator`.

## Translation plugin shape (sketch)

```toml
# samuel-plugin.toml for claude-translator
name = "claude-translator"
version = "1.0.0"
kind = "wasm"

[provides]
hooks = ["after:sync.write-agents-md"]

[hooks.after_sync_write_agents_md]
description = "Mirror AGENTS.md to CLAUDE.md verbatim for Claude Code compatibility"
```

The plugin reads each AGENTS.md the framework wrote and emits a sibling CLAUDE.md. Same content, same autogen marker, same user-customization preservation rule.

## The sync command becomes a hook

User insight: per-folder AGENTS.md generation isn't really a separate user verb — it's something the framework does **at lifecycle points** as a hook.

### Hook points

The sync logic ([[entities/sync-claude-md]]) runs:

- **`samuel init`** → initial sync of every directory.
- **`samuel install <plugin>`** → if the plugin adds language detection, re-sync affected dirs.
- **`samuel run` (per iteration, optional)** → re-sync before agent invocation so directory descriptions stay fresh as the agent creates new dirs.
- **Manual**: `samuel sync` for explicit user-triggered runs (`--dry-run`, `--force`).
- **Optional integrations**: git pre-commit hook, IDE save hook, file watcher. Not built-in; plugin-driven.

### The framework hook definition

```
sync.before              → plugins can pre-process (e.g. detect language from new manifest files)
sync.analyze-folder      → default: heuristic scan; plugin can replace per folder type
sync.write-agents-md     → default: write AGENTS.md
sync.after               → plugin slot: translators run here (CLAUDE.md, Cursor rules, etc.)
```

Each is a named hook point per [[concepts/methodology-default-plus-plugin]]. Plugins register handlers. Default behavior runs if no plugin overrides.

## The `samuel sync` command

Still exists as a top-level verb for manual triggers — useful for:

- Re-running after manually adding folders.
- `--dry-run` preview before committing changes.
- `--force` overwrite of user-customized files.
- CI scripts that regenerate before deploys.

But it's **the same code path** as the hooks. `samuel sync` runs the same `sync.before → analyze → write-agents-md → after` chain that the framework triggers automatically.

## v2 decision #v2-decision

- **Primary file**: AGENTS.md. Cross-tool by default.
- **CLAUDE.md and other tool-specific files**: plugin-driven translations, not framework defaults.
- **Sync as hooks + command**: same code path, two trigger surfaces.
- **Folder generator stays in core**: per-folder AGENTS.md is built-in, part of the Samuel Way ([[concepts/methodology-default-plus-plugin]]).

## What we keep from v1

- The autogen marker (`<!-- Auto-generated by Samuel`).
- The user-customization preservation rule (no marker = skip unless `--force`).
- The heuristic maps (extensions → languages, folder names → purposes). Make user-overridable in `samuel.toml`.

## What we drop

- The forced mirror to CLAUDE.md. v2 writes AGENTS.md only by default.
- The hardcoded path `.claude/skills/...` — v2 paths align with AGENTS.md-primary worldview.

## Open

- Where does CLAUDE.md live when the `claude-translator` plugin is installed? Same directory as AGENTS.md (sibling) seems right. Confirm.
- Should `samuel doctor` warn if AGENTS.md exists but no translator is installed AND the user has Claude Code on PATH? Helpful nudge, or annoying? Tentative: optional check (`--strict`).
- Should `.cursor/rules/` and similar tool-specific dirs be managed by their translator plugins? Yes — the translator owns its tool's filesystem layout, framework doesn't care.
