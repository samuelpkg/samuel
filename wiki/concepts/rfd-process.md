# RFD process

The methodology Samuel uses to document and decide architectural questions. Inherited from [Oxide Computer Company](https://oxide.computer/blog/rfd-1-requests-for-discussion); they inherited it from IETF's RFC tradition.

## What an RFD is

> "Ideas should be timely rather than polished." — Oxide RFD 1

An RFD captures:

- **Why** a decision is being made (problem statement, constraints).
- **What options** were considered (with pros/cons/effort each).
- **What was decided** (chosen option + rationale).
- **What happened** (post-implementation outcome).

It's not a polished spec. It's a thinking artifact. Share early, refine together.

## State machine

```
Prediscussion → Ideation → Discussion → Published → Committed
                    ↓           ↓
                    └─────→ Abandoned
```

| State | Meaning | Location |
|---|---|---|
| **Prediscussion** | Author still forming the idea | `.samuel/rfd/` (private) |
| **Ideation** | Ready for informal feedback | `.samuel/rfd/` (private) |
| **Discussion** | Open for team review | `docs/rfd/` (public) |
| **Published** | Decision made, documented | `docs/rfd/` (public) |
| **Committed** | Implementation complete | `docs/rfd/` (public) |
| **Abandoned** | Rejected or superseded | `docs/rfd/` (public) |

`Abandoned` is **not** failure — preserving "why we did NOT do X" is as valuable as "why we did Y."

## File shape

```markdown
---
rfd: "0042"
title: <one-line title>
authors:
  - name: <handle>
state: Discussion
labels: [architecture, performance, ...]
created: YYYY-MM-DD
updated: YYYY-MM-DD
discussion: <link, optional>
related_prd: <link, optional>
---

# RFD 0042: <title>

## Summary

(1-2 paragraph TL;DR)

## Problem Statement

(What's broken or unclear. Include Requirements + Constraints.)

## Background

(Prior art. Research. Industry patterns. Existing code state.)

## Options Considered

### Option A: <name>

(description)

**Pros:** ...
**Cons:** ...
**Effort:** Low / Medium / High

### Option B: <name>

...

### Option C: <name> (Chosen)

...

## Decision

(Which option. Why. Trade-offs accepted.)

## Implementation Plan

(Phases, milestones, who, when.)

## Outcome

(Post-implementation reflection. What worked, what didn't, what we'd change.)
```

## RFD vs PRD vs Spec

| RFD | PRD |
|---|---|
| Explore options | Define implementation |
| "Why" + "What options" | "What" + "How" |
| Early ideation | Ready to implement |
| Decision documented | Task list generated |

Flow: **Idea → RFD (decide) → PRD (plan) → Tasks → Code**.

RFDs and PRDs together comprise the "Deconstruct + Diagnose" half of [[concepts/4d-methodology]]. Code is the "Develop + Deliver" half.

## Index file

`rfd-index.yaml` (v1) or `rfd-index.toml` (v2 per [[entities/config-format]]) is the master metadata index:

```toml
schema_version = 1
next_number = 9

[[rfds]]
number = "0001"
title = "Three-tier plugin architecture"
state = "Committed"
authors = ["ar4mirez"]
labels = ["plugin", "architecture", "wasm", "oci"]
created = "2026-05-12"
updated = "2026-05-12"
path = "docs/rfd/0001.md"
discussion = ""
related_prd = ""
```

- Single source of truth for state + ownership.
- `next_number` auto-incremented by `create-rfd` plugin.
- Indexing pages (`docs/rfd/index.md`, mkdocs nav) generated from this.

## When to write an RFD

**Yes:**

- Choosing between multiple valid approaches.
- Architectural decisions that affect multiple subsystems.
- Significant changes that need team consensus.
- Decisions whose rationale is non-obvious (future-you will thank past-you).

**No:**

- Obvious decisions with one viable option.
- Small, easily reversible changes.
- Implementation details that fit in a PR description.
- Urgent fixes (document post-hoc if architectural fallout).

## v2 decision #v2-decision

- **RFD process is built into Samuel as the architectural decision pattern.** Not a plugin — it's the Samuel Way.
- **The `create-rfd` plugin** lives in the starter pack ([[synthesis/v2-skill-migration-plan]]). User-facing scaffolding for new RFDs.
- **`rfd-index.toml`** at project root. Generated by the plugin; readable by the framework for nav/search.
- **`docs/rfd/` directory structure** mirrors v1.
- **v2 will use its own RFDs to document its own design.** See [[synthesis/v2-rfds-to-write]] for the inaugural list.

## Open

- **Inter-RFD references.** When RFD 0042 supersedes RFD 0019, both should reference each other. Add `supersedes` and `superseded_by` to the index schema? Probably yes.
- **Discussion link conventions.** v1's frontmatter has a `discussion` field but it's mostly null. Worth tying to GitHub PR / Discussions / issue link explicitly.
- **Abandoned-RFD visibility.** Should `docs/rfd/index.md` list Abandoned RFDs separately, or mix them in? v1 lists Abandoned alongside Committed. Recommendation: separate section ("Abandoned RFDs - kept for reference") so the active set is scannable.
